D4 Error Taxonomy
Signal Failure dominant
Model unable to extract feature signal
Feature additions (interactions, lags, rolling means, city factor) produced near-zero RMSE change under linear regression. Manifested in Exp 4, 5, 7, 8 — all reverted. Root cause was not bad features but a model architecture too weak to capture nonlinear weather/season relationships. Confirmed by Exp 10: same features, RF model, −24% RMSE.
Code Instability
Crashes or broken pipeline
No crashes observed across all 10 runs. Pipeline ran cleanly with run.R as the frozen executor. Residual risk: ranger package availability on new machines requires install check before agent runs. Added to program.md as a pre-run requirement.
Evaluation Leakage
Compromised metric comparability
No leakage detected. Val set (2019–2020) was never used in training. Train/val/test split is time-based and frozen in prepare.R. Kept/reverted decisions are made automatically by run.R against the same val set each run — no manual selection bias.

Codex modified only train.R across all 10 runs. Frozen files (prepare.R, run.R) were untouched. Commits confirm scope was respected: 3 kept commits in Week 4 (b12dc8d, 5553d77, 04dbbaa), 2 reverts via git checkout train.R.
D5 Failure Analysis Memo
Dominant failure mode — signal failure under linear regression

Across 10 experiments, the most persistent failure was not crashes, not evaluation leakage, and not agent misbehavior — it was the model being structurally unable to use the features being added to it. Experiments 4, 5, 7, and 8 all added reasonable, domain-motivated features and were reverted because RMSE did not improve. In each case, the explanation is the same: a linear model cannot capture the nonlinear, cross-city, and temporally-structured relationships in daily solar irradiance data.

Key evidence: Experiment 10 held all features constant (Exp 6 set) and changed only the model family from lm to ranger Random Forest. Val RMSE dropped from 0.9716 to 0.7366 — a 24% single-step improvement. This is the largest gain in the entire experiment log, produced by the cleanest controlled change: one variable, one outcome, one causal account. Exp 10 is the most trusted result in the project.

What this implies about reverted experiments: Features that failed under linear regression — city factor (Exp 4), lag variables (Exp 5), interaction terms (Exp 7), rolling means (Exp 8) — should not be considered bad features. The signal was always there. The linear model lacked the capacity to extract it. Now that Random Forest is the baseline model, each of those features deserves a controlled retest.

Proposed next steps:

1. Rerun Exp 4 (city factor) on top of the Exp 10 RF baseline as a controlled one-variable addition — this is the highest-priority retest given that city encodes climate regime, which RF should exploit heavily.

2. Rerun Exp 5 (lag features) on the RF baseline. Temporal autocorrelation in weather data is a strong signal that tree-based models can use.

3. Begin RF hyperparameter search as a second experiment axis: vary mtry and min.node.size while holding Exp 10 features fixed.

4. After feature retests, consider adding XGBoost as a model comparison to establish whether gradient boosting outperforms bagging on this dataset.
